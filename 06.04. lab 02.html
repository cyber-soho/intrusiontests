Introduction — What Is Passive Reconnaissance?

Passive reconnaissance — commonly called OSINT (Open-Source Intelligence) gathering — is the art of accumulating intelligence about a target using only publicly available, legally accessible data sources, without ever directly contacting or probing the target's infrastructure.

Think of it this way: before a skilled detective ever approaches a suspect, they spend hours in archives cross-referencing public records, analyzing documents, and building a mental map of everything already known. You send nothing. You touch nothing. You leave no trace on the target's systems. Yet by the time you finish, you may know more about the target's digital surface than their own IT department does.

This phase maps directly to Phase 1 (Intelligence Gathering) of the PTES (Penetration Testing Execution Standard) and the reconnaissance stage in the CEH (Certified Ethical Hacker) methodology. Understanding it thoroughly is non-negotiable for every security professional.

P1A

WHOIS Enumeration — ~45 min

WHOIS is a query-and-response protocol standardized in RFC 3912 — the internet's public land registry. A WHOIS record for a poorly-managed domain can reveal a registrant's full name, email address, phone number, physical address, domain creation and expiry dates, registrar identity, name servers, and DNSSEC (DNS Security Extensions) status.

Basic WHOIS Queries

bash
# Basic WHOIS lookup — save full output
whois $TARGET | tee ~/labs/day06-lab02/whois/whois_basic.txt

# Query a specific WHOIS server directly (Verisign handles .com / .net)
whois -h whois.verisign-grs.com $TARGET | tee ~/labs/day06-lab02/whois/whois_verisign.txt

# Resolve the IP address first, then WHOIS the IP
TARGET_IP=$(dig +short A $TARGET | head -1)
echo "Resolved IP: $TARGET_IP"
whois $TARGET_IP | tee ~/labs/day06-lab02/whois/whois_ip.txt

P1B

DNS Enumeration — dig & nslookup — ~60 min

DNS (Domain Name System) is the internet's distributed phonebook. For a penetration tester, DNS records are a treasure map — they reveal mail servers (MX), name servers (NS), IP addresses (A/AAAA), email authentication policies (TXT — SPF, DMARC, DKIM), service declarations (SRV), and canonical aliases (CNAME) that expose third-party services.

TypeFull NameWhat It Reveals
AAddress RecordIPv4 (Internet Protocol version 4) address
AAAAQuad-A RecordIPv6 (Internet Protocol version 6) address
MXMail ExchangerMail servers — critical for email phishing recon
NSName ServerWho controls DNS for this domain
TXTText RecordSPF, DKIM, DMARC, verification tokens — gold mine
SOAStart of AuthorityPrimary NS, admin email, serial, refresh intervals
CNAMECanonical NameAliases — reveals CDN usage, third-party services
SRVService RecordExposes internal services (SIP, XMPP, Lync)
PTRPointer RecordReverse DNS — maps IP back to hostname
bash — Systematic DNS Enumeration
OUTPUT_DIR=~/labs/day06-lab02/dns

# A record (IPv4 — Internet Protocol version 4)
dig A $TARGET +short | tee $OUTPUT_DIR/a_records.txt
dig A $TARGET | tee $OUTPUT_DIR/a_records_full.txt

# AAAA record (IPv6 — Internet Protocol version 6)
dig AAAA $TARGET +short | tee $OUTPUT_DIR/aaaa_records.txt

# MX records (Mail Exchanger — critical for phishing intelligence)
dig MX $TARGET | tee $OUTPUT_DIR/mx_records.txt

# TXT records (Text — SPF / DMARC / DKIM / verification tokens)
dig TXT $TARGET | tee $OUTPUT_DIR/txt_records.txt

# NS records (Name Server)
dig NS $TARGET +short | tee $OUTPUT_DIR/ns_records.txt

# SOA record (Start of Authority — admin email often here)
dig SOA $TARGET | tee $OUTPUT_DIR/soa_record.txt

# Common subdomain CNAME sweep
for sub in www mail remote vpn api dev staging ftp blog admin; do
    echo "--- $sub.$TARGET ---"
    dig CNAME $sub.$TARGET +short
done | tee $OUTPUT_DIR/cname_sweep.txt

# Compare two different public resolvers
dig A $TARGET @8.8.8.8 +short | tee $OUTPUT_DIR/google_resolver.txt
dig A $TARGET @1.1.1.1 +short | tee $OUTPUT_DIR/cloudflare_resolver.txt
// DNS ZONE TRANSFER (AXFR — Authoritative Transfer) — VULNERABILITY TEST

A Zone Transfer is a legitimate DNS replication mechanism. If the primary NS is not restricted to trusted secondary servers only, any host can request a full dump of ALL DNS records. This is a critical misconfiguration that hands an attacker your entire DNS infrastructure on a silver platter.

bash — Zone Transfer Attempt
# Get the authoritative name server first
NS_SERVER=$(dig NS $TARGET +short | head -1)
echo "Testing zone transfer against: $NS_SERVER"

# Attempt AXFR (Authoritative Transfer) — zone transfer
dig AXFR $TARGET @$NS_SERVER | tee ~/labs/day06-lab02/dns/zone_transfer.txt

# If successful → CRITICAL finding. Document fully.
# If blocked  → Expected behavior. Still document the test.
Expected Outcomes : Success → CRITICAL misconfiguration. Document all records received. Blocked → Normal secure behavior; document that the control is in place.

3. Zone transfer attempt — mark SUCCESS or BLOCKED with explanation.

theHarvester — Multi-Source Email & Subdomain Harvesting — ~60 min

theHarvester — developed by Christian Martorella — queries multiple data sources simultaneously: search engines, certificate transparency databases, DNS reverse lookup services, and data aggregators. A single well-targeted run can surface an entire organization's email naming convention, subdomain list, and IP infrastructure.

bash — Search Engine Sources
OUTPUT_DIR=~/labs/day06-lab02/harvester

# Run across search engines (-l = limit results per source)
theHarvester -d $TARGET -b google,bing,duckduckgo -l 500 \
    -f $OUTPUT_DIR/harvester_search.html \
    | tee $OUTPUT_DIR/harvester_search.txt

# Certificate Transparency (CT) sources — often the richest for subdomains
theHarvester -d $TARGET -b crtsh,certspotter,dnsdumpster,rapiddns \
    -f $OUTPUT_DIR/harvester_ct.html \
    | tee $OUTPUT_DIR/harvester_ct.txt

# FULL comprehensive multi-source run (may take 10-15 minutes — document the wait)
theHarvester -d $TARGET \
    -b google,bing,duckduckgo,crtsh,certspotter,dnsdumpster,hackertarget,rapiddns,urlscan,virustotal \
    -l 1000 \
    -f $OUTPUT_DIR/harvester_full.html \
    | tee $OUTPUT_DIR/harvester_full.txt

# Open the HTML (HyperText Markup Language) report in a browser
firefox $OUTPUT_DIR/harvester_full.html &

What to Extract & Document

  • List every email address discovered. Identify the naming convention (e.g. firstname.lastname@domain.com)
  • List all subdomains. Run dig A on each to see which resolve.
  • Map IP addresses to cloud providers (AWS — Amazon Web Services, Azure, GCP — Google Cloud Platform)
  • Note any hosts with open ports returned by Shodan-enabled sources

P2B

Amass — Deep Subdomain Enumeration — ~45 min

Amass — an OWASP (Open Web Application Security Project) project — is a depth specialist for DNS and subdomain data. It queries certificate transparency logs, DNS datasets, scrapes search engines, and cross-references threat intelligence sources. Use passive mode only for this lab.

bash — Passive Subdomain Enumeration
OUTPUT_DIR=~/labs/day06-lab02/amass

# PASSIVE mode only — no direct contact with target
# This can take 15-30 minutes. Let it run. Document the duration.
amass enum -passive -d $TARGET \
    -o $OUTPUT_DIR/amass_subdomains.txt \
    -json $OUTPUT_DIR/amass_output.json \
    | tee $OUTPUT_DIR/amass_terminal.txt

echo "Unique subdomains found:"
wc -l $OUTPUT_DIR/amass_subdomains.txt

# Parse JSON (JavaScript Object Notation) — extract unique names and data sources
cat $OUTPUT_DIR/amass_output.json | python3 -c "
import sys, json
names = set()
sources = set()
for line in sys.stdin:
    try:
        e = json.loads(line.strip())
        names.add(e.get('name',''))
        for s in e.get('sources',[]): sources.add(s)
    except: pass
print('=== UNIQUE SUBDOMAINS ===')
for n in sorted(names): print(n)
print(f'\n=== SOURCES USED ({len(sources)}) ===')
for s in sorted(sources): print(s)
"

# Cross-reference: what did Amass find that theHarvester missed ?
grep -oP "[a-zA-Z0-9._-]+\.$TARGET" ~/labs/day06-lab02/harvester/harvester_full.txt \
    | sort -u > ~/labs/day06-lab02/harvester_subs_only.txt

comm -23 \
    <(sort $OUTPUT_DIR/amass_subdomains.txt) \
    <(sort ~/labs/day06-lab02/harvester_subs_only.txt) \
    | tee $OUTPUT_DIR/amass_exclusive.txt

echo "Subdomains ONLY in Amass (missed by theHarvester):"
cat $OUTPUT_DIR/amass_exclusive.txt

P3A

Shodan — Intelligence on Exposed Services — ~45 min

Shodan is the world's first search engine for Internet-connected devices, created by John Matherly in 2009. Unlike Google, Shodan continuously scans the entire public IPv4 (Internet Protocol version 4) address space and indexes banners — raw service responses — of every open port it discovers. This reveals software versions, operating systems, SSL certificate details, and even default credential warnings — without you touching the target directly.

Shodan Web Interface Queries

shodan search syntax — web UI at shodan.io
# Search by hostname
hostname:acmecorplab.xyz

# Search by SSL (Secure Sockets Layer) certificate — excellent subdomain discovery
ssl.cert.subject.cn:acmecorplab.xyz

# Filter by port — look for exposed admin panels, SSH (Secure Shell), etc.
hostname:acmecorplab.xyz port:22
hostname:acmecorplab.xyz port:443
hostname:acmecorplab.xyz port:8080
hostname:acmecorplab.xyz port:3389

# Search for vulnerable software versions (example)
hostname:acmecorplab.xyz product:Apache http.component.version:2.4.49

Shodan CLI (Command Line Interface) — Programmatic Queries

bash — Shodan CLI
OUTPUT_DIR=~/labs/day06-lab02/shodan

# Verify API (Application Programming Interface) access and credits
shodan info | tee $OUTPUT_DIR/shodan_account.txt

# Get target IP then query Shodan host database
TARGET_IP=$(dig +short A $TARGET | head -1)
shodan host $TARGET_IP | tee $OUTPUT_DIR/shodan_host.txt

# Search by hostname — structured output
shodan search --fields ip_str,port,org,os hostname:$TARGET \
    | tee $OUTPUT_DIR/shodan_hostname.txt

# Search by SSL certificate — finds additional subdomains !
shodan search --fields ip_str,port,org ssl.cert.subject.cn:$TARGET \
    | tee $OUTPUT_DIR/shodan_ssl.txt

# Download full JSON (JavaScript Object Notation) results for deep analysis
shodan download $OUTPUT_DIR/shodan_results hostname:$TARGET
shodan parse --fields ip_str,port,org,data $OUTPUT_DIR/shodan_results.json.gz \
    | tee $OUTPUT_DIR/shodan_parsed.txt

Red Flag Checklist — Complete This in Your Report

  • Default credentials warning found
  • Outdated software with known CVEs (Common Vulnerabilities and Exposures)
  • Admin panels exposed to the public Internet
  • Development or staging servers accessible publicly
  • Non-standard ports with interesting services
  • Self-signed SSL (Secure Sockets Layer) certificates suggesting misconfiguration
  • Servers on port 80 with sensitive-looking paths, not redirecting to HTTPS
  • SSL certificate Subject Alternative Names revealing unknown subdomains

Maltego CE — Visual Intelligence Mapping — ~60 min

Maltego is the de-facto standard for visual link analysis in OSINT (Open-Source Intelligence) and cyber threat intelligence. It uses entities (domains, IPs, email addresses, people, organizations) and transforms (automated data-retrieval operations) to build interactive relationship graphs. The Community Edition (CE) is free and fully functional for this lab.

Transforms in Maltego CE query external third-party databases — this remains passive (you are not contacting the target). Do NOT run active transforms. Do NOT target real external organizations. Register free at maltego.com/get-started.
