01

DNS / WHOIS

The Internet's Public Phone Book

What Is DNS (Domain Name System)?

DNS translates human-readable domain names into machine-readable IP (Internet Protocol) addresses. When you type example.com into a browser, a DNS resolver returns 93.184.216.34. But DNS stores far more than IP addresses — each record type reveals a different piece of the target's infrastructure.

RecordWhat It RevealsIntel Value
A / AAAAIPv4 / IPv6 addressHosting provider, CDN (Content Delivery Network)
MXMail Exchanger serverEmail infrastructure
NSName ServerDNS hosting provider
TXTSPF (Sender Policy Framework), DKIM (DomainKeys Identified Mail), DMARC (Domain-based Message Authentication)Third-party services in use
CNAMEAlias to another domainCloud services, CDNs
SOAZone admin email, serial numberAdmin contact, update frequency

What Is WHOIS?

WHOIS is a query-and-response protocol defined in RFC 3912 that lets anyone look up who registered a domain, when it was registered, when it expires, which registrar was used, and what name servers are authoritative. The data is maintained by Regional Internet Registries (RIRs) such as ARIN (American Registry for Internet Numbers) for North America, RIPE NCC for Europe, and APNIC for Asia-Pacific.

Since the introduction of GDPR (General Data Protection Regulation) in 2018, many registrars redact personal details, but corporate registrations still frequently expose the company name, technical contact, and registrar abuse contact.

Real-Life Example — The Coffee-Shop Recon

You are sitting in a coffee shop with your laptop. Your client, "BeanBrew Coffee," has authorised a penetration test. Your first step — before touching any scanner — is to run whois beanbrew.com and dig beanbrew.com ANY. Within seconds you discover their hosting provider (AWS), mail server (Google Workspace via MX records), and that their WHOIS record reveals the CTO's email address. Zero packets sent to the target.

Python 3 Automates DNS record queries and WHOIS lookups 
# Import socket for DNS resolution via the OS resolver
import socket
# Import subprocess to run external commands (whois, nslookup)
import subprocess
# Import sys for command-line argument parsing
import sys
# Import datetime to timestamp the recon report
from datetime import datetime

def resolve_domain(domain):
    # Query the A record via the default DNS resolver
    try:
        ip = socket.gethostbyname(domain)
        print(f"[+] A Record: {domain} → {ip}")
        return ip
    except socket.gaierror:
        print(f"[-] Could not resolve {domain}")
        return None

def whois_lookup(domain):
    # Execute the system 'whois' command and capture output
    result = subprocess.run(
        ["whois", domain],
        capture_output=True, text=True, timeout=15
    )
    print(result.stdout)

def dns_records(domain):
    # Query A, MX, NS, and TXT record types
    for rtype in ["A", "MX", "NS", "TXT"]:
        result = subprocess.run(
            ["nslookup", f"-type={rtype}", domain],
            capture_output=True, text=True, timeout=10
        )
        print(f"--- {rtype} Records ---")
        print(result.stdout)

# Main: parse target, run all recon modules
target = sys.argv[1]
resolve_domain(target)
whois_lookup(target)
dns_records(target)

Step-by-Step Explanation

  1. resolve_domain() — Uses socket.gethostbyname() to look up the A record via the OS resolver.
  2. whois_lookup() — Spawns the whois command, captures registrant data, name servers, and dates.
  3. dns_records() — Loops through A, MX, NS, TXT record types using nslookup.
  4. MX records reveal mail servers; NS records reveal DNS hosting; TXT records contain SPF and verification tokens.

DNS and WHOIS are your absolute first move — they cost nothing, leave no trace on the target, and immediately reveal domain ownership, hosting providers, and mail infrastructure.

Subdomains

Finding the Forgotten Doors

Why Subdomains Are the Skeleton Keys of Passive Recon

Large enterprises can have thousands of subdomains — many created for temporary projects, staging environments, or internal tools, then forgotten. These "shadow" subdomains often run outdated software, lack WAF (Web Application Firewall) protection, and sit on the public internet with default credentials. Each subdomain is a potential entry point.

Passive Discovery Sources

CT (Certificate Transparency) logs are the single most powerful passive source. Since 2018, all major CAs (Certificate Authorities) publish every TLS (Transport Layer Security) certificate to publicly auditable logs. If staging.secret.acme.com ever had a certificate — it is permanently searchable via crt.sh.

Other sources include search-engine dorking (site:target.com), DNS aggregation APIs (SecurityTrails, VirusTotal), and the Wayback Machine.

Real-Life Example — The Forgotten Staging Server

Your client mentions "about five public subdomains." You run a CT log search and discover 47 — including staging.client.com, jenkins.client.com, and old-api.client.com. The staging server runs a two-year-old version with debug mode enabled and a default admin password.

Bash Queries crt.sh CT logs and resolves discovered subdomains 
#!/usr/bin/env bash
# Read target domain from argument; default to example.com
TARGET="${1:-example.com}"

# Query crt.sh Certificate Transparency logs
# curl -s: silent mode; jq: extract domain names; sort -u: deduplicate
curl -s "https://crt.sh/?q=%25.${TARGET}&output=json" \
  | jq -r '.[].name_value' 2>/dev/null \
  | sort -u > "subdomains_${TARGET}.txt"

# Count unique subdomains found
COUNT=$(wc -l < "subdomains_${TARGET}.txt" | tr -d ' ')
echo "[+] Found ${COUNT} unique subdomain(s)"

# Resolve each subdomain to check if it's live
while IFS= read -r sub; do
    # dig +short returns only the IP address
    IP=$(dig +short "$sub" 2>/dev/null | head -1)
    if [ -n "$IP" ]; then
        echo "  ${sub}  →  ${IP}"
    fi
done < "subdomains_${TARGET}.txt"

Step-by-Step Explanation

  1. crt.sh query — Uses curl to fetch JSON from crt.sh with a wildcard query. jq extracts domain names.
  2. Deduplication — sort -u removes duplicate entries and alphabetises.
  3. Resolution loop — For each subdomain, dig +short checks if it has a live A record.
  4. Two output files are produced: all discovered subdomains, and only the live ones with IPs.

Organisations routinely underestimate their own attack surface. Certificate Transparency logs reveal subdomains the target doesn't even know they have — including forgotten staging, dev, and admin servers.

The Credential Reuse Chain

When employees reuse their corporate email and password on third-party services that later get breached, those credentials end up in publicly searchable databases. A penetration tester can find them passively and legally. The most well-known aggregator is Have I Been Pwned (HIBP), created by Troy Hunt. Other sources include DeHashed and IntelligenceX (IntelX).

HIBP13+ billion accounts, 800+ breachesFree / API key

Real-Life Example — The Recycled Password

You query HIBP and discover the CFO (Chief Financial Officer)'s corporate email appeared in the 2021 LinkedIn breach with leaked passwords. During the active phase, you find the same password on the corporate VPN — instant remote access from a single reused credential.

Python 3Queries HIBP API for breach exposure
import requests, sys, time

HIBP_URL = "https://haveibeenpwned.com/api/v3/breachedaccount/"
HEADERS = {"User-Agent": "CyberSoHo-BreachChecker/1.0"}

def check_breach(email):
    # Send GET request to HIBP API with target email
    resp = requests.get(f"{HIBP_URL}{email}?truncateResponse=false", headers=HEADERS)
    if resp.status_code == 200:
        return resp.json()  # Email found in breach(es)
    elif resp.status_code == 404:
        return None         # Not in any known breach
    elif resp.status_code == 429:
        time.sleep(2)        # Rate-limited, pause and retry
        return "RATE_LIMITED"

# Display breach results with name, date, and leaked data classes
result = check_breach(sys.argv[1])
if result:
    for b in result:
        print(f"Breach: {b['Name']} | Date: {b['BreachDate']}")
        print(f"  Leaked: {', '.join(b['DataClasses'])}")

Step-by-Step Explanation

  1. Sends an HTTP GET to HIBP's v3 API with the email address appended to the URL.
  2. HTTP 200 = email found in breach (returns JSON). HTTP 404 = clean. HTTP 429 = rate-limited.
  3. Displays breach names, dates, affected account counts, and leaked data classes (passwords, IPs, etc.).
  4. Note: HIBP API now requires an API key — obtain one at haveibeenpwned.com/API/Key.

Breach data is the highest-value passive OSINT source. The most dangerous vulnerability is often a human one — password reuse. Always check HIBP before active testing begins.

Metadata

The Invisible Fingerprints in Every File

What Is Metadata?

Every document, image, and presentation carries invisible passengers — data about the data. A PDF (Portable Document Format) might reveal the author's Windows username and an internal file path (\\CORPFS01\Finance\Reports\). A photograph could contain GPS (Global Positioning System) coordinates. This is completely passive — download a public file, run ExifTool, and the intelligence flows.

Metadata FieldIntelligence Value
Author / CreatorUsernames → potential login credentials
CompanyConfirms organisation name as configured in Office
Application & VersionSoftware version → matching CVEs (Common Vulnerabilities and Exposures)
File PathInternal server names and folder structure
GPS CoordinatesPhysical office locations from photos
Last Modified ByAdditional user accounts beyond the creator
Real-Life Example — The CEO's Photo Reveals the HQ

You download the CEO's headshot from the company's "About Us" page. Running exiftool ceo_photo.jpg reveals EXIF (Exchangeable Image File Format) data with GPS coordinates pointing to an undisclosed secondary office in Arlington, Virginia, the camera model (iPhone 14 Pro), and the exact date the photo was taken.

Python 3Extracts metadata from PDF, DOCX, and images
import os, sys, subprocess, zipfile
import xml.etree.ElementTree as ET

def extract_with_exiftool(filepath):
    # Run ExifTool — industry standard for metadata extraction
    result = subprocess.run(
        ["exiftool", filepath],
        capture_output=True, text=True, timeout=30
    )
    print(result.stdout)

def extract_office_metadata(filepath):
    # DOCX/PPTX/XLSX are ZIP archives with XML metadata
    with zipfile.ZipFile(filepath, 'r') as zf:
        # Parse docProps/core.xml for author, dates
        if 'docProps/core.xml' in zf.namelist():
            root = ET.fromstring(zf.read('docProps/core.xml'))
            # Extract dc:creator, cp:lastModifiedBy, etc.
        # Parse docProps/app.xml for Application, Company, Version
        if 'docProps/app.xml' in zf.namelist():
            root = ET.fromstring(zf.read('docProps/app.xml'))

# Detect file type and run appropriate extraction
ext = os.path.splitext(sys.argv[1])[1].lower()
if ext in ['.docx', '.pptx', '.xlsx']:
    extract_office_metadata(sys.argv[1])
extract_with_exiftool(sys.argv[1])

Step-by-Step Explanation

  1. ExifTool handles 400+ file formats and is the industry standard for forensic metadata work.
  2. Office XML parsing — opens DOCX/PPTX/XLSX as ZIP archives and reads the XML metadata files directly.
  3. The script extracts author, company, application version, creation dates, and file paths.
  4. Look for: usernames, network paths, GPS coordinates, software versions.

Every published file is a potential intelligence source. A single PDF can reveal usernames, software versions, internal server names, and physical locations — all without the target knowing.

Attack Surface Mapping

Connecting All the Dots Into a Single Picture

What Is an Attack Surface?

The attack surface is the totality of every digital asset an organisation exposes — every system, service, port, API (Application Programming Interface) endpoint, employee email, cloud storage bucket, and third-party integration. The wider the attack surface, the more opportunities an adversary has. Attack surface mapping combines all data from Lessons 1–4 into a structured inventory.

The Mapping Process

1. Collect — DNS records, subdomains, breach data, metadata. 2. Catalogue — Organise every asset by type and risk. 3. Resolve — Map subdomains to IPs, identify hosting providers. 4. Enrich — Cross-reference with Shodan and Censys for open ports and service banners. 5. Visualise — Create a unified map with Maltego or a custom tool. 6. Prioritise — Rank by risk to guide active testing.

ToolFunctionBest For
ShodanInternet-wide device & service searchOpen ports, banners, IoT devices
CensysHost and certificate searchTLS certificate analysis
MaltegoVisual link-analysis graphConnecting entities visually
SpiderFootAutomated OSINT collectionComprehensive automation
Recon-ngModular OSINT frameworkCustom recon workflows
Real-Life Example — The Forgotten S3 Bucket

Mapping a healthcare startup, you discover an open Elasticsearch instance on port 9200 via Shodan with no authentication — containing patient appointment logs. A DNS TXT record reveals an AWS S3 verification token. You find healthcorp-backups.s3.amazonaws.com is publicly readable — containing database exports with PHI (Protected Health Information) that violates HIPAA.

Attack surface mapping is not about finding one vulnerability — it is about understanding the shape of the target's total digital exposure so you can prioritise where to focus active testing.

Threat Modeling

From Data Collection to Strategic Thinking

The STRIDE Framework

Created by Microsoft engineers Loren Kohnfelder and Praerit Garg in 1999, STRIDE provides a systematic checklist ensuring no major threat category is overlooked. For each component in a Data Flow Diagram (DFD), you ask all six questions.

S

Spoofing

Can an attacker pretend to be someone or something else? (e.g., forging email headers to impersonate the CEO)

T

Tampering

Can data be modified without authorisation? (e.g., altering a price field in an HTTP request)

R

Repudiation

Can an actor deny having performed an action? (e.g., deleting audit logs after unauthorised access)

I

Information Disclosure

Can sensitive data leak to unauthorised parties? (e.g., error messages revealing database table names)

D

Denial of Service

Can an attacker disrupt availability? (e.g., flooding a login page with requests)

E

Elevation of Privilege

Can an attacker gain higher access than allowed? (e.g., a regular user reaching admin endpoints)

The Four-Step Process

1. Identify Assets — What are we protecting? 2. Map Components — Draw a DFD (Data Flow Diagram) with trust boundaries. 3. Apply STRIDE — Ask all six questions per component. 4. Prioritise & Mitigate — Rate risks and propose countermeasures.

Other methodologies: PASTA (Process for Attack Simulation and Threat Analysis), DREAD (Damage, Reproducibility, Exploitability, Affected Users, Discoverability), and LINDDUN (privacy-focused). Free tools include OWASP Threat Dragon and the Microsoft Threat Modeling Tool.

Real-Life Example — Banking STRIDE Analysis

Testing an online banking portal, you STRIDE-model four components: web app, REST API, MySQL database, and payment processor. The API analysis immediately reveals: (S) tokens can be spoofed without proper signing, (T) transaction amounts are vulnerable to tampering, (I) verbose errors disclose stack traces, and (E) horizontal privilege escalation lets one customer view another's account.
